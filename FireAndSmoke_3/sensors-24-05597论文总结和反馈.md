## sensors-24-05597论文总结和反馈

下面我按“**论文做了什么—解决什么问题—创新点—实验结果—研究方向进展—局限与启发**”来分析。这篇论文的核心不是提出一个全新的 YOLO 网络结构，而是**构建一个更贴近真实场景的火灾/烟雾开放数据集，并系统比较多代 YOLO 检测器在该数据集上的表现**。

# 一、论文基本信息与研究定位

论文题目是 **A Comparative Performance Evaluation of YOLO-Type Detectors on a New Open Fire and Smoke Dataset**，发表在 *Sensors* 2024 年第 24 卷。研究对象是基于视觉传感器图像的**火焰与烟雾目标检测**，属于智能视觉感知、目标检测、灾害预警、智慧城市安全监测方向。论文提出了一个名为 **FireAndSmoke** 的开放数据集，包含超过 22,000 张图像和 93,000 个实例，图像来自 1200 多个 YouTube 视频及公开网络资源，并覆盖白天/夜间、室内/室外、城市/工业/森林、低/高分辨率、单目标/多目标等复杂场景。

这篇文章的研究逻辑可以概括为一句话：**真实火灾场景复杂、传统传感器误报多、现有视觉数据集不够全面，因此作者构建一个更复杂的数据集，并比较 YOLOv5 到 YOLOv10、YOLO-NAS 等模型，找出更适合实时火灾烟雾检测的模型。**

# 二、这篇论文主要解决了什么问题？

## 1. 传统火灾报警系统误报和漏报问题

传统火灾报警主要依赖烟雾传感器、温度传感器、红外传感器等，但这些传感器在大范围、开放环境、复杂工业场景中存在明显不足。例如烟雾探测器难以区分真实火灾烟雾与灰尘、水汽、厨房油烟、香烟烟雾等干扰源。论文中提到，自动触发火警并不一定都是真实火灾，且大量误报与气溶胶、灰尘、水汽有关。

因此，作者希望通过计算机视觉方式，让系统不仅“感知到异常”，还可以**定位火焰/烟雾区域**，并输出置信度、检测框和实时处理速度，这比单一传感器报警更适合视频监控、无人机巡检、智慧城市、森林防火等场景。

## 2. 现有火灾/烟雾数据集不够贴近真实复杂环境

论文指出，已有公开数据集在类别、场景、天气、分辨率、拍摄角度、传感器来源等方面差异很大。有的数据集只检测火焰，有的只检测烟雾，有的只有火/非火二分类，有的只针对森林火灾或船舶火灾，泛化能力容易受限制。

作者认为，如果模型只看“典型火焰”和“典型烟雾”，很容易在真实场景中把夕阳、车灯、路灯、广告屏、反光、云层等误判为火焰或烟雾。因此，他们在数据集中专门加入了 **other 类别**，用于标注“看起来像火或烟，但实际上不是火灾”的干扰目标。这个设计是论文解决误报问题的关键。

## 3. 不同 YOLO 模型缺乏统一数据集下的公平比较

很多论文都说自己的模型效果好，但往往是在不同数据集、不同类别定义、不同实验条件下测试，横向比较很困难。本文把 YOLOv5、YOLOv6、YOLOv7、YOLOv8、YOLOv9、YOLOv10 和 YOLO-NAS 放到同一个 FireAndSmoke 数据集上，用 Precision、Recall、mAP 和 FPS 统一评估。

所以它解决的另一个问题是：**为火灾烟雾检测方向提供一个相对统一的 benchmark。**

# 三、论文的技术路线

论文整体流程可以分成四步。

第一步是数据采集。作者从 YouTube 视频和开放网络图像中收集火焰、烟雾以及容易误判的干扰图像。数据覆盖建筑火灾、车辆火灾、森林火灾、垃圾燃烧、室内火灾、无人机航拍、远距离小目标、夜间火光等多种场景。

第二步是数据预处理。图像被统一调整为模型需要的尺寸，并使用直方图均衡来缓解光照变化影响，同时使用水平翻转、±20°旋转等增强方式扩展样本多样性。

第三步是半自动标注。作者先人工标注 1000 张图像，然后训练 YOLOv8m 模型，用它对剩余图像进行自动预测，再由人工检查、修正边界框、删除错误标注，最终保证数据集质量。这个过程比完全人工标注更省力，又比纯自动标注更可靠。

第四步是模型比较。作者在同一数据集上训练和测试七类 YOLO 模型，比较它们对 fire、smoke、other 三类目标的检测效果，并进一步在白天和夜间混合视频中测试模型实时检测能力。

# 四、论文创新点分析

## 创新点 1：构建了新的开放 FireAndSmoke 数据集

这是本文最核心的贡献。数据集不仅包含火焰和烟雾，还包含大量容易误判为火灾的 **other 类**，比如夕阳、云、路灯、车灯、反光、应急灯、电子屏光源等。这个设计直接针对真实部署中的误报问题。

相比很多只关注“火/非火”或“火/烟”的数据集，这个数据集更接近实际监控场景，因为现实中真正困难的往往不是识别一团明显火焰，而是判断“这个亮光到底是不是火”。

## 创新点 2：场景覆盖更广，增强模型泛化测试价值

数据集包括白天 15,812 张图像、夜间 7158 张图像，涉及烟雾-only、火焰-only、火焰+烟雾、无人机航拍、森林/草地/野火，以及夜间车灯、广告牌、街灯等干扰目标。

这使得模型不仅要学会“火焰长什么样”，还要学会在尺度、光照、距离、遮挡、背景复杂度变化下稳定检测。

## 创新点 3：提出了半自动标注流程

作者没有完全手工标注全部图像，而是先标注小样本，再训练 YOLOv8m 生成初始检测框，之后人工审核和修正。这种流程在工程上很实用，因为大规模目标检测数据集最耗时的部分就是标注。

这个贡献虽然不是算法结构创新，但对数据集构建非常重要，尤其适合后续你如果做遥感目标检测、多源图像融合数据集整理，也可以借鉴这种“少量人工标注 + 模型预标注 + 人工质检”的路线。

## 创新点 4：系统比较了 YOLOv5 到 YOLOv10 及 YOLO-NAS

本文比较的模型覆盖较新，尤其包含 YOLOv9、YOLOv10 和 YOLO-NAS。论文介绍了不同 YOLO 模型的特点，例如 YOLOv8 的 anchor-free 检测、多尺度特征图和 Distribution Focal Loss，YOLOv9 的 PGI 与 GELAN，YOLOv10 的 NMS-free 训练和一致双分配策略，YOLO-NAS 的神经架构搜索思想。

这使得论文不仅是一个数据集论文，也有模型选择参考价值。

# 五、实验结果与结论分析

从测试集结果看，YOLOv8 和 YOLOv10 整体表现较优，火焰和烟雾类别的 mAP@50 最高可达到约 0.91，说明模型对“是否存在火焰/烟雾”的检测能力已经较强。论文摘要中也指出，YOLOv10 是综合表现最好的模型，准确率约 89%，mAP@50 超过 91%。

但如果看更严格的 mAP@50–95，数值明显下降到 0.60–0.70 左右，这说明模型虽然能检测到火焰和烟雾，但在边界框定位精度上仍有提升空间。尤其烟雾边界本身模糊、形态变化大，精确框选比检测火焰更难。

模型差异可以这样理解：

| 模型     | 主要特点                             | 适合场景                     |
| -------- | ------------------------------------ | ---------------------------- |
| YOLOv5   | 速度快、表现稳定                     | 实时监控、轻量部署           |
| YOLOv7   | 检测数量较多                         | 宁可多报也不能漏报的场景     |
| YOLOv8   | 综合平衡较好                         | 一般火灾监控场景             |
| YOLOv9   | 参数多，速度较慢                     | 对精度有要求但算力充足       |
| YOLOv10  | 最稳健，误报控制较好                 | 消防应急、工业安全、远程报警 |
| YOLO-NAS | 对远距离小火点有一定优势，但速度较慢 | 小目标火焰检测、非强实时场景 |

论文讨论部分指出，YOLOv10 在尺度变化、光照变化、复杂烟雾背景下表现最稳健，尤其适合需要降低误报的应用；YOLOv6 和 YOLOv7 检测数量更多，适合大范围工业园区、森林区域等“漏检成本高”的场景；YOLOv5 和 YOLOv10 在白天/夜间视频中都有较高置信度检测能力。

# 六、这个研究方向取得了什么进展？

从论文梳理来看，火灾/烟雾视觉检测方向经历了几个阶段。

早期主要依赖手工特征，比如颜色空间、纹理、形状、运动特征、光流、灰度共生矩阵等。这些方法可解释性较强，但对光照、天气、背景复杂度非常敏感。论文也指出，手工特征方法复杂且劳动密集，面对低质量图像、恶劣天气、复杂背景时效果受限。

之后进入深度学习阶段，CNN、Faster R-CNN、YOLO、Transformer 等方法开始用于火灾烟雾检测。深度学习能够自动学习火焰和烟雾的视觉特征，相比传统方法减少了人工设计特征的依赖，并提升了检测精度和实时性。

当前阶段的进展重点不只是“模型更大更准”，而是更强调：

一是**实时性**。火灾检测必须快，YOLO 系列因为速度快、精度高，成为该方向常用模型。

二是**复杂环境泛化能力**。真实场景有夜间、雾天、雨天、低分辨率、远距离、小目标、遮挡、强光干扰等问题。

三是**误报控制**。other 类样本、难例样本、对比学习、可解释性分析都会成为后续重点。

四是**工程部署**。论文已经提到将 YOLO 火灾检测模型集成到 NVIDIA DeepStream 视频处理框架中，通过 5G 网络传输摄像头画面，在服务器端生成报警和可视化面板，用于消防员和应急服务人员。

所以，这个方向已经从“单张图像分类是否有火”发展到“复杂视频场景下实时定位火焰/烟雾，并服务于应急响应系统”。

# 七、论文不足与可继续研究的点

这篇论文有价值，但也有明显局限。

第一，它主要是数据集和模型评估论文，没有提出新的检测网络结构。也就是说，算法创新性不算特别强，贡献更偏向数据集构建和 benchmark。

第二，虽然加入了 other 类降低误报，但真实世界中的干扰远远不止车灯、夕阳、云层等，还包括工业火花、焊接光、电弧、雾霾、蒸汽、沙尘、雨雪、摄像头过曝等，未来仍需要更复杂的难例数据。

第三，实验主要基于可见光图像。如果用于真实火灾预警，后续可以结合红外、热成像、烟雾传感器、温湿度传感器、风速风向、地理位置等多源信息，这就与“多源异构传感器数据融合”方向更接近。

第四，论文提到 XAI 可解释人工智能是未来方向，因为目标检测不仅要给出类别，还要给出位置和置信度，解释难度高于普通分类。未来如果能解释模型为什么把某个亮光判定为火焰，就更有利于真实部署和消防系统信任。

第五，作者也指出后续会考虑 Transformer、集成学习、对比学习，以及雨、雾、雪等恶劣天气下的检测效果。

# 八、总结归纳

这篇论文的主要贡献可以归纳为：**面向真实火灾烟雾检测场景，作者构建了一个包含 fire、smoke、other 三类目标的大规模开放数据集 FireAndSmoke，并通过半自动标注方式提高数据构建效率；随后在统一实验条件下比较 YOLOv5–YOLOv10 与 YOLO-NAS 的性能，证明 YOLOv10 在复杂光照和尺度变化条件下具有较好的稳健性，而 YOLOv5 速度较快、YOLOv7 检测覆盖更高、YOLO-NAS 对远距离小目标有一定优势。**

放到研究方向里看，它推动了火灾烟雾检测从“简单分类/单一场景检测”向“复杂真实场景、多类别干扰抑制、实时目标检测、工程部署验证”发展。它对你做目标检测、多源传感器融合、遥感/无人机灾害监测方向都有参考价值，尤其是“难例样本构建”“误报类别设计”“多模型 benchmark 比较”这几个思路很值得借鉴。



## 最终解决问题的建议和反馈

原作者的 FireAndSmoke 数据集本身强调三点：**多场景覆盖、fire/smoke/other 三类、day/night 与远近尺度变化**。论文中也明确把 other 类作为区分夕阳、车灯、路灯、反光、电子屏等易误报干扰源的重要类别，并且原实验中 YOLOv8/YOLOv10 在测试集上表现相对较好。 原仓库 README 也强调该数据集包含建筑火、车辆火、植被火、野火、垃圾火、室内火、无人机视角、远景火焰，以及 fire/smoke/other 三类设计。

下面给你一套比较适合你当前复现实验的**改进方案**。核心目标不是“再训练一个模型”，而是建立一个完整闭环：**数据清洗 → 标注统一 → 难例增强 → 小目标训练 → 切片推理 → 视频时序后处理 → 分场景评估**。

------

# 1. 先明确你的问题本质

你现在说的第一个现象：

> 大场景里多处火焰/烟雾同时出现时，模型不能全部识别。

这个问题通常来自 5 个原因：

第一，**标注方式不统一**。有的图把一整片火焰框成一个大框，有的图只框局部火苗，模型就会不知道到底应该输出一个大框，还是多个小框。

第二，**密集目标训练样本不足**。如果训练集中“一张图多个独立火点/烟点”的样本比例不够，模型推理时就容易只抓最显著的火点，忽略远处、暗处、小尺寸火焰。

第三，**640 输入尺寸压缩太狠**。大场景视频缩放到 640 后，远处火焰可能只剩几个像素，YOLO 的特征层很难保留有效信息。

第四，**NMS 抑制掉相邻框**。如果多处火焰距离近、框有重叠，NMS 的 IoU 阈值设置不合适，会把一些正确框压掉。

第五，**置信度阈值太高**。很多小火点、早期烟雾本来置信度就低，如果 conf 设置 0.4 或 0.5，很容易被过滤掉。AllYoloModels 仓库的推理脚本本身支持 `conf_thr`、`count`、`filter`、`nrCompareFrames`、`boxSimilarity` 等参数，这些正好可以用于阈值扫描和视频错误分析。

所以你现在要做的不是单纯“换模型”，而是先让数据和推理策略匹配你的实际场景。

------

# 2. 第一阶段：建立标准测试集，不要只靠视频肉眼判断

你已经多轮测试了，但建议先把测试集固定下来。否则每次换模型、换参数、换数据，都很难判断到底哪里提升了。

建议建立 5 个子测试集：

| 测试集                      | 内容                                 | 主要看什么                    |
| --------------------------- | ------------------------------------ | ----------------------------- |
| Test-A 原 Sensors 测试集    | 原作者数据，不加入自有数据           | 检查是否接近原论文/原仓库结果 |
| Test-B 自有 fire_pic 测试集 | 你自己的真实场景                     | 检查实际应用泛化能力          |
| Test-C 大场景小目标         | 远处火点、无人机、监控远景           | 检查小目标 Recall             |
| Test-D 多火点/多烟雾        | 一帧内多处火焰、烟雾                 | 检查多实例召回                |
| Test-E other 硬负样本       | 车灯、路灯、夕阳、反光、广告屏、雾气 | 检查误报率                    |

每个测试集不要太大，先做精标版本即可。比如每类 200–500 张图，关键是**标注必须非常一致**。你后面所有模型都用这 5 个测试集评估。

不要只看 mAP，还要看这些指标：

| 指标                  | 意义                          |
| --------------------- | ----------------------------- |
| Fire Recall           | 火焰有没有漏检                |
| Smoke Recall          | 烟雾有没有漏检                |
| Other Precision       | 干扰物有没有误报成火/烟       |
| Small Object Recall   | 远处小火点召回率              |
| Multi-instance Recall | 一张图多个火点时，匹配到几个  |
| FP/min                | 视频每分钟误报次数            |
| Detection Delay       | 视频中火/烟出现后几帧被检测到 |

论文原实验主要报告 Precision、Recall、mAP、FPS 等指标；但你现在的问题是“多目标漏检、小目标漏检、视频场景不稳”，所以必须额外加 **Small Object Recall、Multi-instance Recall、FP/min、Detection Delay**。

------

# 3. 第二阶段：重新统一标注规范，这是最关键的一步

你说“数据变多不等于模型变强”，这句话非常对。尤其火焰和烟雾这种非刚性目标，标注风格对模型影响很大。

建议你制定一个固定标注规范，之后所有数据都按这个规则修。

## 3.1 fire 类标注规范

建议这样定：

**fire 框的是可见火焰主体，不框烟雾，不框纯光晕，不框反光。**

具体规则：

| 情况                     | 标注方式                                          |
| ------------------------ | ------------------------------------------------- |
| 一片连续燃烧区域         | 框成一个 fire                                     |
| 多个明显分离火点         | 分别标多个 fire                                   |
| 火焰和烟雾同时存在       | fire 和 smoke 分开框                              |
| 夜间火焰过曝形成大光团   | 只框火源主体，不把整片光晕全框进去                |
| 车灯、警灯、广告屏、夕阳 | 标 other，不标 fire                               |
| 反射出的火光             | 如果不是实际火焰，标 other 或不标，按你的规则固定 |

最容易出问题的是“一个大框 vs 多个小框”。建议你的规则是：**物理上分离的火点分开标；视觉上连续燃烧的火区合并标。**

例如一栋楼三扇窗户分别着火，如果火焰中间断开，就标 3 个 fire；如果已经连成一片，就标 1 个 fire。

## 3.2 smoke 类标注规范

烟雾比火焰更难。建议规则：

**smoke 框的是可见烟羽主体，不要把大片天空、背景、雾气全部框进去。**

| 情况               | 标注方式                                        |
| ------------------ | ----------------------------------------------- |
| 浓烟边界清晰       | 紧贴烟雾主体框                                  |
| 淡烟边界模糊       | 框可见烟雾核心区域，不无限扩大                  |
| 烟雾被风吹成长条   | 框主要连续烟羽                                  |
| 烟雾分成多团       | 分开标多个 smoke                                |
| 云、雾、蒸汽、灰尘 | 如果像烟但非火灾，标 other 或加入 hard negative |
| 火焰上方烟雾       | fire 和 smoke 分开标                            |

烟雾不要框得太松。框得太松，模型会学到“天空 + 一点灰色 = smoke”；框得太紧，模型又容易只看局部纹理。建议你固定为：**框住肉眼可辨认烟雾主体，允许少量背景，但不要超过目标区域 10%–20% 太多。**

## 3.3 other 类标注规范

other 类是你第二轮训练没有提升的关键点之一。

原作者的 other 类不是普通背景，而是**容易被误判为火/烟的难负样本**，包括日出日落、云、路灯、车灯、反射、应急灯、电子屏等。论文和仓库都反复强调了这个类别。 

你现在加入大量 fire/smoke，但没有同步加入 other，模型会变得“更敏感”，结果就是：召回可能没明显提高，误报反而增加。

建议 other 至少补这些类型：

| other 类型        | 对应误报风险               |
| ----------------- | -------------------------- |
| 夜间车灯          | 误报 fire                  |
| 消防车/警车灯     | 误报 fire                  |
| 路灯/广告屏       | 误报 fire                  |
| 夕阳/日出         | 误报 fire                  |
| 玻璃反光/金属反光 | 误报 fire                  |
| 云层/雾气/蒸汽    | 误报 smoke                 |
| 工业烟囱白汽      | 误报 smoke                 |
| 焊接火花/电弧光   | 误报 fire                  |
| 烟花/篝火/舞台灯  | 视应用决定标 other 或 fire |

建议比例上，不要让 other 太少。原论文训练集中 fire、smoke、other 实例大致是 19864、18017、10648，other 约为 fire 的一半以上。 你现在如果 fire/smoke 翻倍，但 other 没跟上，3 类模型的判别边界一定会偏。

# 4. 第三阶段：不要直接混合数据，先做数据体检

你现在的 Sensors + fire_pic 合并到 4.3 万张，这一步很可能引入了分布偏移。建议先做数据统计脚本，把两个数据源分别分析。

你需要统计这些内容：

| 统计项               | 为什么重要                       |
| -------------------- | -------------------------------- |
| 每类实例数量         | 看 fire/smoke/other 是否失衡     |
| 每张图目标数量       | 看是否缺少多实例样本             |
| bbox 面积分布        | 看小目标比例是否够               |
| bbox 宽高比分布      | 看烟雾框是否过松                 |
| 每个数据源的类别比例 | 看自有数据是否压过 Sensors       |
| day/night 比例       | 看夜间是否被削弱                 |
| 场景类别比例         | 建筑火、车辆火、森林火、室内火等 |
| 空标注图片比例       | 看负样本是否足够                 |
| 重复图片/近重复帧    | 防止视频帧过度重复               |

尤其要看 bbox 面积分布。你可以按 COCO 常用思路粗略分：

| 类型   | bbox 面积占图像比例 |
| ------ | ------------------- |
| small  | < 1%                |
| medium | 1%–10%              |
| large  | > 10%               |

如果你的大场景漏检严重，大概率是 small fire / small smoke 的样本太少，或者训练时目标被缩放后太小。

建议你不要一上来就混合训练，而是做 4 个版本：

| 版本 | 数据组成                             | 目的                 |
| ---- | ------------------------------------ | -------------------- |
| D0   | 原 Sensors                           | 复现原作者基线       |
| D1   | 原 Sensors + 自有原始 fire_pic       | 验证直接混合是否变差 |
| D2   | 原 Sensors + 统一标注后的 fire_pic   | 验证标注统一收益     |
| D3   | D2 + other 硬负样本 + 小目标切片样本 | 目标版本             |

如果 D1 比 D0 差，说明不是模型问题，是数据合并方式有问题。
 如果 D2 明显好于 D1，说明标注规范是主因。
 如果 D3 好于 D2，说明小目标和 other 补强有效。

# 5. 第四阶段：对“大场景小目标”单独处理

你已经准确指出：**imgsz=640 对大场景小目标限制明显。**

这类问题不要指望单纯继续训练解决，建议采用两条路线同时做。

## 路线 A：提高训练和推理输入尺寸

先尝试：

| 模式           | imgsz |
| -------------- | ----- |
| 快速对比       | 640   |
| 推荐训练       | 960   |
| 小目标增强     | 1280  |
| 离线高精度测试 | 1536  |

如果你的显卡显存够，优先试 `imgsz=960` 或 `imgsz=1280`。Ultralytics 的 YOLO 推理支持 `conf`、`iou`、`imgsz`、`device` 等参数，可以直接在预测阶段扫描不同输入尺寸。

建议你做一个固定测试：

| imgsz | conf | iou  | 观察           |
| ----- | ---- | ---- | -------------- |
| 640   | 0.25 | 0.6  | 当前基线       |
| 960   | 0.25 | 0.6  | 小目标是否提升 |
| 1280  | 0.20 | 0.7  | 多目标是否提升 |
| 1536  | 0.15 | 0.7  | 离线极限召回   |

如果 1280 明显提升，说明问题主要是分辨率，不是模型结构。

## 路线 B：切片检测 / SAHI

对于大场景远处火点，切片检测非常重要。SAHI 的核心思路就是把高分辨率图像切成重叠小块分别检测，再把结果合并；相关论文也指出，小目标在图像中像素太少时是常规检测器的主要难点，而 slicing aided inference 可以提升小目标检测 AP，并且可以叠加 fine-tuning 继续提升。

建议参数从这里开始：

| 参数       | 推荐值                |
| ---------- | --------------------- |
| slice_size | 640 或 768            |
| overlap    | 0.20–0.30             |
| conf       | fire 0.18，smoke 0.12 |
| merge      | class-wise NMS 或 WBF |
| iou merge  | 0.5–0.6               |

你的实际视频中，如果是 1920×1080，可以切成 640×640 或 768×768；如果是 4K 视频，切片几乎是必须的。

推荐最终推理方式：

**全图检测一次 + 切片检测一次 + 合并结果。**

全图检测负责大火、大烟和整体上下文；切片检测负责远处小火点、小烟雾。两者结合比单独 640 输入强很多。

# 6. 第五阶段：训练方式建议

不要直接用 4.3 万张混合数据重新训。建议采用“三阶段训练”。

## 阶段 1：原 Sensors 基线复现

目标：确认你的代码、环境、参数没有偏。

原论文训练使用 100 epoch，不同 YOLO 模型 batch size 不同，YOLOv5m、YOLOv8m、YOLOv10m 等都在同一数据集上比较。 原仓库也说明训练通过 AllYoloModels 中的 `YoloModelsTrain.sh` 脚本完成，并使用参数文件设置数据路径和测试视频。

你先不要加入自有数据，只训练 Sensors，得到一个 baseline。
 如果 baseline 都比原作者差很多，先查环境、数据版本、训练参数、类别顺序、yaml 路径、标注格式。

重点检查：

| 检查项         | 常见错误                        |
| -------------- | ------------------------------- |
| class 顺序     | fire/smoke/other 顺序错会直接崩 |
| labels 坐标    | YOLO 格式归一化是否正确         |
| train/val/test | 是否路径错、是否空标签          |
| weights        | 是否从正确预训练权重开始        |
| imgsz          | 是否和原实验一致                |
| batch          | 是否过小导致 BN/训练不稳        |
| 数据增强       | 是否和原作者差异太大            |

## 阶段 2：Sensors → 自有数据微调，而不是直接混合

如果你的目标是实际场景，可以这样：

先用 Sensors 训练一个泛化能力强的基础模型，再用自有数据微调，但微调时不要只喂自有数据。建议混合比例：

| 数据           | 比例 |
| -------------- | ---- |
| Sensors        | 50%  |
| 自有 fire_pic  | 30%  |
| other 硬负样本 | 20%  |

这样可以避免自有数据把 Sensors 的多场景能力“冲掉”。

训练策略：

| 参数         | 建议                                     |
| ------------ | ---------------------------------------- |
| 初始权重     | Sensors baseline best.pt                 |
| 学习率       | 原训练的 1/5 到 1/10                     |
| epoch        | 30–60                                    |
| imgsz        | 960 或 1280                              |
| mosaic       | 前期保留，后期关闭                       |
| close_mosaic | 最后 10–20 epoch                         |
| patience     | 20–30                                    |
| freeze       | 可先冻结 backbone 5–10 epoch，再全量训练 |

这个阶段的目标不是追求训练集 mAP，而是让模型适应你的真实视频场景。

## 阶段 3：小目标切片 fine-tuning

把大场景视频帧切片生成训练数据。

方法：

1. 从大场景视频中抽帧。
2. 对整图做标注。
3. 自动切成 640/768 patch。
4. 保留包含 fire/smoke 的 patch。
5. 随机保留一部分不含火烟但含 other 的 patch。
6. 生成 sliced training dataset。
7. 用 Sensors baseline 或阶段 2 模型继续 fine-tune。

这样做的好处是：原来在整图中只有 10×10 像素的小火点，切片后可能变成相对更大的目标，模型更容易学习。

# 7. 第六阶段：推理参数不要用固定默认值

你现在测试视频时，不要只用一个 conf。建议做阈值扫描。

## 7.1 conf 阈值建议

火焰和烟雾的置信度阈值应该分开。

| 类别  | 建议 conf |
| ----- | --------- |
| fire  | 0.18–0.30 |
| smoke | 0.10–0.25 |
| other | 0.25–0.40 |

烟雾早期低对比，conf 必须比 fire 低。如果统一设置 0.4 或 0.5，烟雾召回一定差。

但是 conf 降低会带来误报，所以必须配合视频时序过滤。也就是说：

**单帧低阈值召回，多帧一致性抑制误报。**

## 7.2 NMS IoU 建议

多火点漏检时，检查 NMS。

| 场景       | iou 建议  |
| ---------- | --------- |
| 普通场景   | 0.5–0.6   |
| 多火点密集 | 0.65–0.75 |
| 小目标密集 | 0.7 左右  |
| 误报太多   | 0.5 左右  |

如果 iou 太低，相邻火焰框可能被压掉。
 如果 iou 太高，重复框会变多。
 你的目标是“多处火焰不能漏”，可以先把 iou 提到 0.65–0.7 观察 Recall 是否提升。

## 7.3 推荐推理组合

先用下面几组跑你的固定视频：

| 方案 | imgsz                   | conf      | iou  | 适合         |
| ---- | ----------------------- | --------- | ---- | ------------ |
| P0   | 640                     | 0.25      | 0.6  | 当前基线     |
| P1   | 960                     | 0.20      | 0.65 | 推荐实时     |
| P2   | 1280                    | 0.18      | 0.70 | 大场景小目标 |
| P3   | sliced 640 overlap 0.25 | 0.15      | 0.60 | 高召回       |
| P4   | full 960 + sliced 640   | 0.15–0.20 | 0.65 | 最终方案     |

如果 P2/P3 明显好于 P0，就说明你主要瓶颈是输入分辨率和小目标，而不是模型本身。

# 8. 第七阶段：视频时序后处理，尤其针对早期烟雾

你说早期烟雾难，这是事实。YOLO 是单帧检测，它没有天然的时间判断能力。烟雾的“慢慢扩散、持续变浓、方向漂移”是视频时序特征，单帧 YOLO 学不到。

建议加一个轻量时序模块：

## 8.1 多帧确认机制

规则：

- fire：连续 2/5 帧检测到，触发有效目标；
- smoke：连续 3/8 帧检测到，触发有效目标；
- other：如果某区域长期稳定不动，且形态不扩散，降低火/烟置信度。

这样可以允许你把单帧 conf 降低，提高召回，同时不让误报爆炸。

## 8.2 简单跟踪

对每个检测框建立 track：

| track 信息      | 用途                 |
| --------------- | -------------------- |
| bbox 坐标       | 判断目标是否连续存在 |
| class           | fire/smoke/other     |
| confidence 均值 | 平滑置信度           |
| 持续帧数        | 判断是否报警         |
| 面积变化        | 烟雾是否扩散         |
| 位置漂移        | 烟雾是否随风移动     |

不一定要复杂算法，先用 IoU 匹配就够。
 如果后面想更稳，可以接 ByteTrack 或 BoT-SORT。

## 8.3 烟雾专门规则

烟雾可以加一些启发式特征：

| 特征                   | 烟雾倾向       |
| ---------------------- | -------------- |
| 区域面积逐渐扩大       | 是             |
| 边缘模糊               | 是             |
| 颜色低饱和、灰白/黑灰  | 是             |
| 连续向上或随风漂移     | 是             |
| 与火焰区域空间相邻     | 是             |
| 位置完全固定、亮度稳定 | 更像灯/云/背景 |

这部分不需要替代 YOLO，而是作为后处理加权。

最终可以这样融合：

```
final_score = YOLO_conf × 0.6 + temporal_score × 0.3 + motion_score × 0.1
```

对早期烟雾尤其有用。

# 9. 第八阶段：建议你最终采用“两路检测 pipeline”

我建议你不要只依赖单模型单尺度推理，而是采用下面这个结构：

```
输入视频帧
   ↓
全图 YOLO 检测 imgsz=960/1280
   ↓
大场景判断：如果分辨率高、目标小、检测数少
   ↓
切片 YOLO 检测 640/768 overlap=0.25
   ↓
合并 full-frame + sliced detections
   ↓
class-wise NMS / WBF
   ↓
时序跟踪与多帧确认
   ↓
输出 fire / smoke / other + 报警等级
```

报警等级可以分成：

| 等级    | 条件                                   |
| ------- | -------------------------------------- |
| L1 可疑 | 单帧低置信 fire/smoke                  |
| L2 关注 | 连续多帧出现                           |
| L3 警报 | fire 高置信或 smoke 持续扩散           |
| L4 严重 | 多处 fire/smoke 同时出现或面积快速扩大 |

这样你的系统会比单纯 YOLO 框图更像一个真正可部署的火灾检测系统。

# 10. 模型选择建议

你现在训练出来两个模型，如果效果不理想，不建议马上换一堆模型。优先选 2–3 个做深度优化。

根据论文和原仓库结果，推荐这样选：

| 模型                | 用途                           |
| ------------------- | ------------------------------ |
| YOLOv10m / YOLOv10b | 主模型，稳定性好，误报控制较好 |
| YOLOv8m / YOLOv8l   | 工程易用，训练和部署方便       |
| YOLOv5m / YOLOv5l   | 对比基线，速度快，生态成熟     |
| YOLO-NAS            | 可作为小目标对照，但实时性较弱 |

论文讨论中提到 YOLOv10 在尺度和光照变化下整体更稳健，YOLOv5 推理速度快，YOLO-NAS 对远距离小火点有一定优势。 但你的目标不是论文复现排名，而是实际视频准确预测，所以优先建议：

**主线：YOLOv10m/b + imgsz 960/1280 + 切片推理 + 时序后处理。**

如果部署压力大：

**轻量线：YOLOv8m + imgsz 960 + 切片只在大场景触发。**

# 13. 最推荐的最终方案

你的目标是“模型准确预测”，我建议最终方案定为：

```
数据层：
Sensors 原数据
+ 统一标注后的自有 fire_pic
+ 与 fire/smoke 数量匹配的 other hard negative
+ 大场景小目标切片数据
+ 多火点/多烟雾密集样本

训练层：
YOLOv10m/b 或 YOLOv8m/l
imgsz=960/1280
先 Sensors baseline
再混合微调
最后 sliced fine-tuning

推理层：
full-frame imgsz=960/1280
+ sliced inference 640/768 overlap=0.25
+ class-wise NMS / WBF
+ fire/smoke 分类别阈值
+ 视频多帧确认与简单 tracking

评估层：
mAP50/mAP50-95
+ Small Object Recall
+ Multi-instance Recall
+ FP/min
+ Detection Delay
+ day/night 分场景统计
```

一句话总结：**不要再单纯扩大数据集；你现在应该把重点放在“标注一致性、other 硬负样本、小目标切片训练/推理、视频时序后处理”这四件事上。**

最优先做的 3 件事是：

1. **把自有 fire_pic 按统一规范重标一部分，先做高质量小样本，不追求数量。**
2. **补充 other 硬负样本，尤其是夜间车灯、路灯、夕阳、反光、云雾、蒸汽。**
3. **对大场景视频使用 imgsz=1280 和切片推理测试，先验证小目标召回是否明显提升。**
